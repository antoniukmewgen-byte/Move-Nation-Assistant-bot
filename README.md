# MoveNation Assistant Bot

Бот + Mini App для керування клієнтськими групами команди MoveNation. Групи створюються та клієнти додаються від імені того співробітника, який ініціює дію — через його власну MTProto-авторизацію (Telethon), а не від спільного userbot-акаунта.

## Компоненти
- **app/bot** — aiogram-бот: вибір посади, `/connect` (авторизація особистого акаунта), створення групи, додавання клієнтів, теги, /team, /register
- **app/bot/guards.py** — узгоджене звуження Optional-полів aiogram (`from_user`, `text`, `message` у callback'ах) в одному місці
- **app/userbot/actions.py** — Telethon-функції, що створюють тимчасовий клієнт на базі збереженої сесії конкретного співробітника: реальне створення груп у Telegram, лого, додавання учасників
- **app/services/crypto.py** — шифрування/розшифрування збережених сесій (Fernet)
- **app/services/telegram_auth.py** — серверна HMAC-валідація Telegram WebApp `initData` (єдине джерело довіри для ідентифікації користувача Mini App)
- **app/api** — FastAPI backend для Mini App, захищений `initData`-авторизацією та перевіркою членства в групі
- **app/services/reminders.py** — фонові нагадування кожні 5 хв
- **app/logging_config.py** — централізоване логування (консоль + ротація файлів у `logs/`)
- **miniapp/** — статичний Mini App (форма створення групи, список команди з тегами)
- **alembic/** — міграції схеми БД
- **tests/** — pytest-тести для CRUD-шару та валідації `initData`

## Запуск (локально, без Docker)

1. Створити віртуальне середовище і встановити залежності:
   ```
   python -m venv .venv
   .venv\Scripts\activate
   pip install -r requirements.txt
   ```

2. Скопіювати `.env.example` у `.env` і заповнити:
   - `BOT_TOKEN` — токен від @BotFather для `MoveNation_Assistant_bot`
   - `API_ID` / `API_HASH` — з https://my.telegram.org (API development tools)
   - `BOT_USERNAME` — username бота без `@` (потрібен, щоб бот сам додавався адміном у нові групи)
   - `SESSION_ENCRYPTION_KEY` — ключ для шифрування збережених сесій користувачів. Згенерувати:
     ```
     python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
     ```
     Значення перевіряється при старті застосунку (`app/config.py`) — некоректний ключ впаде одразу з поясненням, а не десь у рантаймі під час шифрування сесії.
   - `CORS_ALLOWED_ORIGINS` — залиш порожнім, якщо Mini App віддається з того самого FastAPI-застосунку (типовий випадок)
   - `INIT_DATA_MAX_AGE_SECONDS` / `LOG_LEVEL` — опційні, мають розумні дефолти

3. Покласти дефолтне лого групи у `assets/default_logo.png`.

4. Застосувати міграції БД (детальніше — розділ «Міграції БД» нижче). Це **обов'язковий крок** — застосунок більше не створює таблиці сам при старті, щоб не було розсинхрону зі схемою в Alembic:
   ```
   python -m alembic upgrade head
   ```
   Папку `data/` створювати вручну не потрібно — вона й сам файл БД зʼявляються автоматично.

5. Запустити:
   ```
   python -m app.main
   ```

6. Кожен співробітник, який має створювати групи чи додавати клієнтів, один раз авторизує свій особистий Telegram-акаунт через `/connect` у чаті з ботом (номер телефону → код → пароль 2FA за потреби). Сесія зберігається у БД у зашифрованому вигляді і використовується лише для дій, які сам користувач ініціює через бота або Mini App.

## Запуск через Docker

```
cp .env.example .env   # і заповнити значення
docker compose up --build
```

Контейнер при старті сам застосовує міграції (`alembic upgrade head`), а потім запускає бота + API + планувальник нагадувань в одному процесі (`CMD` у `Dockerfile`). Дані БД і логи монтуються в `./data` та `./logs` на хості, тож переживають перестворення контейнера. Healthcheck опитує `GET /health`.

## ⚠️ Лише один інстанс

Цей застосунок **не** розрахований на горизонтальне масштабування — має бути запущений рівно один процес/контейнер одночасно. Ніколи не роби:
- `docker compose up --scale bot=2` (або будь-який `deploy.replicas > 1`)
- Kubernetes/Swarm deployment з `replicas > 1`
- запуск `python -m app.main` вручну паралельно з уже працюючим інстансом (напр. другий локальний процес поруч із Docker-контейнером)

Причини — обидві структурні, не про продуктивність:

1. **aiogram long-polling.** `dp.start_polling(bot)` у `app/main.py` викликає Telegram Bot API `getUpdates`. Два паралельні інстанси з тим самим `BOT_TOKEN` почнуть «перетягувати» одні й ті самі апдейти одне в одного (Telegram віддає апдейт лише одному з конкурентних `getUpdates`-запитів), і повідомлення користувачів непередбачувано губитимуться або дублюватимуться.
2. **In-memory стан авторизації.** `app/services/telethon_auth.py` тримає `_pending_clients` / `_pending_phone_data` (проміжний стан `/connect`: телефон → код → 2FA) просто в dict у пам'яті процесу — свідомо, без Redis чи БД (див. докстрінг модуля). Якщо запит коду обробить інстанс A, а введений код прилетить на інстанс B (типово при round-robin балансувальнику), інстанс B нічого не знайде — користувач побачить «Сесія авторизації втрачена» на рівному місці.

Це не обмеження продуктивності, яке колись треба буде «зняти масштабуванням» — весь бот свідомо спроєктований як один asyncio-процес (`app/main.py`: один `asyncio.run(main())`, один `uvicorn.Server` без `workers=`, спільна aiogram FSM `MemoryStorage`). Якщо колись знадобиться кілька інстансів (напр. для HA), для цього спершу потрібно винести обидва in-memory стани в спільне сховище (Redis) і перейти на Telegram-вебхуки замість `getUpdates`-polling — це вже інша архітектура, а не конфіг-прапорець.

## Міграції БД (Alembic)

Схема БД версіонується виключно через Alembic (`alembic/`) — це єдине джерело істини. Застосунок (`app/main.py`) свідомо **не** викликає `Base.metadata.create_all()` при старті: якщо схему можна створити двома незалежними шляхами (сирими метаданими і трекованими міграціями), рано чи пізно вони розійдуться або Alembic відмовиться накатувати міграцію на таблиці, яких «не пам'ятає». Тому без `alembic upgrade head` таблиць просто не буде.

```
alembic upgrade head                       # застосувати всі міграції
alembic revision --autogenerate -m "..."   # згенерувати нову міграцію за змінами в app/db/models.py
alembic downgrade -1                       # відкотити останню міграцію
```

`alembic/env.py` бере `DATABASE_URL` напряму з `app.config.settings`, тож окремо налаштовувати `alembic.ini` не потрібно — досить валідного `.env`.

### Rollback runbook (production-інцидент)

Якщо після деплою нова міграція зламала прод (застосунок не стартує, помилки на кожен запит до БД, дані пишуться в неправильну колонку тощо) — послідовність дій:

1. **Зупини трафік до застосунку**, якщо є така можливість (напр. `docker compose stop bot`), щоб він не продовжував писати в БД, поки схема неконсистентна.

2. **Зроби бекап поточного файлу БД перед будь-якими діями**, навіть якщо він уже «зламаний» — це єдина точка повернення, якщо відкат теж піде не так:
   ```
   cp data/*.db data/backup-before-rollback-$(date +%Y%m%d-%H%M%S).db
   ```
   (шлях/ім'я файлу — за `DATABASE_URL` у `.env`; для контейнера — те саме всередині змонтованого `./data`).

3. **Перевір, чи downgrade() у проблемної міграції взагалі написаний і безпечний.** Alembic автогенерує `downgrade()` за структурними змінами (додати/видалити колонку тощо), але **не** відновлює дані, які `upgrade()` міг трансформувати чи видалити (напр. `DROP COLUMN` незворотно губить дані цієї колонки — `downgrade()` поверне лише порожню колонку). Відкривай файл міграції в `alembic/versions/` і читай `downgrade()` очима, перш ніж запускати.

4. **Відкоти рівно на одну міграцію назад** (не більше, якщо не впевнений, що знаєш, що робиш):
   ```
   alembic downgrade -1
   ```
   Щоб відкотитися до конкретної відомої-робочої ревізії:
   ```
   alembic history                 # знайти revision id, на якому все ще було добре
   alembic downgrade <revision_id>
   ```

5. **Перевір поточний стан схеми** перед тим, як знову пускати трафік:
   ```
   alembic current                 # яка ревізія застосована зараз
   alembic check                   # чи відповідає модель у app/db/models.py поточній БД (Alembic ≥ 1.13)
   ```

6. **Заверни назад код застосунку**, якщо проблемна міграція йшла в парі з релізом, який очікує нову схему (напр. новий код звертається до колонки, якої після відкату вже нема) — сам по собі `alembic downgrade` без відповідного відкату коду часто ламає щось інше.

7. **Запусти трафік знову** і одразу перевір `GET /health` (`{"status": "ok", "database": true}`) та логи (`logs/`) на предмет нових помилок.

8. **Пиши новий forward-fix, а не «латай» стару міграцію.** Якщо стара міграція вже застосовувалась на проді (навіть якщо потім відкочена), не редагуй її файл заднім числом — інші середовища (dev/staging), які вже накотили її як є, розійдуться. Замість цього створи нову ревізію (`alembic revision --autogenerate -m "..."`), яка виправляє проблему.

**Важливо:** усе вище стосується тільки downgrade *схеми*. Якщо міграція вже встигла необоротно змінити чи видалити дані (не тільки структуру), сам Alembic це не відновить — єдиний надійний шлях назад у такому разі — бекап БД, зроблений *до* деплою (звідси й обов'язковий крок 1/2: зупинити трафік і забекапити поточний стан ще до того, як почнеш щось відкочувати).

## Безпека Mini App

Mini App більше не довіряє `user_id`, який передає клієнт у тілі запиту чи query-рядку — це давало б змогу видати себе за будь-якого співробітника. Замість цього:

- Фронтенд (`miniapp/js/api.js`) надсилає сирий підписаний `Telegram.WebApp.initData` у заголовку `X-Telegram-Init-Data` з кожним запитом.
- Бекенд (`app/services/telegram_auth.py`) перевіряє HMAC-SHA256 підпис за офіційним алгоритмом Telegram і вік `auth_date`, і лише тоді довіряє `user_id` з нього (`app/api/deps.py`).
- Кожен ендпоінт, що працює з конкретною групою (`app/api/routes/groups.py`, `members.py`), додатково перевіряє, що верифікований користувач є учасником цієї групи (`crud.user_is_group_member`), закриваючи IDOR-и на кшталт перегляду чужого списку учасників за відомим `group_id`.
- CORS увімкнено, лише якщо явно задано `CORS_ALLOWED_ORIGINS` — за замовчуванням Mini App same-origin і додаткові заголовки не потрібні.
- Помилки серверної логіки (Telethon, БД) логуються повністю (`logger.exception`), а клієнту повертається узагальнене повідомлення — деталі виключення більше не «протікають» назовні.

## Розробка та якість коду

Дев-залежності (включно з тестовими та lint-інструментами):
```
pip install -r requirements-dev.txt
```

**Лінтер і форматер (Ruff):**
```
ruff check .            # лінт
ruff check . --fix      # автофікс, де можливо
ruff format .           # форматування
ruff format --check .   # перевірка форматування без змін
```

**Типізація (Mypy):**
```
mypy
```

**Тести (Pytest, з in-memory SQLite):**
```
pytest                              # усі тести
pytest --cov=app --cov-report=term-missing   # з покриттям
```

**Компіляція (швидка перевірка на синтаксичні помилки):**
```
python -m compileall -q app
```

Усі ці перевірки виконуються автоматично в CI (`.github/workflows/ci.yml`) на кожен push у `main` та кожен pull request, разом з перевіркою, що Alembic-міграції застосовуються й відкочуються без помилок.

## Нотатки
- Privacy Mode бота в @BotFather має бути вимкнено (`/setprivacy` → Disable), інакше бот не бачитиме повідомлення клієнтів у групах.
- Для старих груп: додай бота вручну, зроби адміністратором, виконай `/register`, познач учасників через `/tag` (у відповідь на повідомлення) або через Mini App.
- Дії через Mini App (створення групи, додавання клієнта) також вимагають, щоб відповідний користувач заздалегідь пройшов `/connect` у боті — інакше API поверне помилку 400.
- Логи пишуться в консоль і в `logs/` (ротація по 5 МБ, до 5 файлів); рівень регулюється `LOG_LEVEL` в `.env`.
- `/health` повертає `{"status": "ok"|"degraded", "database": bool}` — використовується Docker healthcheck-ом і придатний для зовнішнього моніторингу.
